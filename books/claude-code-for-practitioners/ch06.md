---
title: "MCP を使うか自前 Python にするかの判断軸"
---

## 結論を先に書く

外部サービス連携の手段として MCP(Model Context Protocol)と自前 Python が選べる場面で、私は次の比率で使い分けています。

- **MCP を使う**: 対話的な作業、認可フローが必要な操作、公式メンテが期待できるサービス
- **自前 Python を使う**: scheduled-tasks などのヘッドレス自動化、コストを抑えたい場合、運用責任を自分で持ちたいサービス

割合としては、私の個人インフラの中で 80% は自前 Python、20% が MCP です。本章はその判断根拠を書きます。

---

## MCP の強み

MCP は Model Context Protocol の略で、Claude(と他の AI)が外部サービスを「共通の作法で」叩くための標準プロトコルです。Notion、Slack、Google Calendar、Linear、ファイルシステム、Web ブラウザ等、対応サービスは増え続けています。

### 強み 1: 対話作業に強い

Claude との対話中に「Notion のこのページを更新して」「Slack でチームに送って」と言ったとき、MCP サーバーが既に接続されていれば即座に動きます。自前 Python では「スクリプトを書いて実行する」一手間が入ります。

### 強み 2: 認可フローを公式が面倒見てくれる

OAuth トークンの取得・更新、API 呼び出しの形式、エラーハンドリングが MCP サーバーに閉じています。自前 Python で書くと、毎回認証の管理を自分で書く必要があります。

### 強み 3: アップデートを公式がフォロー

外部 API の仕様変更にも、MCP サーバー側がアップデートで吸収してくれることが多い。自前 Python は仕様変更のたびに自分で直す必要があります。

---

## MCP の弱み

### 弱み 1: 対話前提のサーバーが多い

ブラウザ系 MCP(Chrome 制御など)や、対話的に認証するサーバーは、scheduled-tasks のような無人実行で使えません。実行中にユーザーの入力を求められるからです。

私の運用では、scheduled-tasks の登録時に「このジョブは認可ダイアログを出すかもしれません」と Claude から言われたら、それは MCP 依存です。自前 Python に切り替える検討対象になります。

### 弱み 2: 依存が増える

MCP サーバーが落ちると、それを使う全 skill が止まります。自前 Python なら、`subprocess.run(["security", "find-generic-password", ...])` のような OS 標準コマンドへの依存だけで動きます。

### 弱み 3: コストが見えにくい

MCP サーバー経由の操作も、最終的には Claude API のトークン消費に乗ります。サーバー側がどれだけのコンテキストを Claude に渡しているかは透明ではないため、コスト最適化が難しい場面があります。

---

## 自前 Python の強み

### 強み 1: ヘッドレス対応

scheduled-tasks、cron、launchd 等のあらゆるスケジューラから呼べます。認証はトークン直接呼び出しで完結します。

### 強み 2: コスト透明

API 呼び出しは自分で書くため、何にいくらかかっているかが見えます。Claude API を呼ばないジョブ(ただの Web スクレイピング)なら、API コストはゼロです。

### 強み 3: 運用責任が自分にある

これは強みでもあり弱みでもあります。仕様変更があれば自分で直すしかないが、逆に言えば「直せば動く」状態を維持できます。MCP のメンテが止まると、自分では何もできません。

---

## 自前 Python の弱み

### 弱み 1: 認証コードを毎回書く

OAuth フロー、トークン更新、エラーハンドリングを毎回実装する必要があります。私は notion-workspace skill で `subprocess.run(["security", ...])` で Keychain から Notion トークンを取り出すパターンを書いて、それを他 skill から import するようにしています。

### 弱み 2: 仕様変更を自分で追う

Notion API は 2025-09-03 で大きな破壊的変更がありました(`database_id` → `data_source_id` への移行)。これに気づかなければ skill が止まります。自前 Python では、こうした変更を自分で監視する必要があります。

### 弱み 3: ボイラープレートが多い

エラーハンドリング、リトライ、ロギング、テストを全部自分で書きます。MCP サーバーなら一部はサーバー側に任せられます。

---

## 私の判断ルール

私の運用ルールは次の通りです。

- **Notion へのデータ投入は自前 Python**: scheduled-tasks から呼ぶため、また Keychain 連携を統一したいため
- **Notion でのページ閲覧・編集の対話作業は MCP**: 「このページの中身を見て」「ここを更新して」と Claude に言うときは MCP
- **Slack 通知は MCP**: 認可が面倒なため
- **Web スクレイピング(HN、dev.to、Anthropic News)は自前 Python**: API キーが要らないため、コストゼロ
- **メール操作は AppleScript**: そもそも Claude を呼ばない、launchd で十分

新しい外部サービスを追加するときは、上記ルールに当てはめて MCP / 自前 Python のどちらで始めるかを最初に決めます。決めずに着手すると、両方半端なまま放置されます。

---

## 章末演習

自分が連携している(または連携したい)外部サービスを 5 つリストアップして、次の 3 つの問いに答えてください。

- 対話的に呼ぶか / 自動実行で呼ぶか
- 認可フローを自分で書けるか / 書きたくないか
- 仕様変更を自分で追えるか / 公式に任せたいか

3 つのうち 2 つ以上が「自分で」側なら自前 Python、「公式に任せる」側なら MCP。同点なら自前 Python(運用責任を自分で持つほうが長期的に強い)。
