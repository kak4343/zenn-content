## 自動化には 2 種類しかない

個人インフラの自動化を分類すると、結局のところ 2 種類に収束します。

- **イベント駆動**: 何かをきっかけに動く(ユーザーの言葉、ファイルの変化、メール受信)
- **時間駆動**: 時計で動く(毎朝 6 時、毎週月曜、毎月 1 日)

Claude Code では前者が **skill のキーワード発動**、後者が **scheduled-tasks** に対応します。本章ではこの 2 つの使い分けを、設計レベルで整理します。

---

## skill のキーワード発動を支配するのは `description`

`~/.claude/skills/<name>/SKILL.md` の frontmatter にある `description` フィールドが、その skill の「発動条件」です。Claude はユーザーの発言を読み、各 skill の description と照合し、最も合致するものを呼び出します。

私の `baito-management` skill の description を抜粋すると、こんな具合です(冗長に見えますが意図的です):

```yaml
description: |
  医師の外勤・非常勤バイトを Notion + iPhone 主体で一元管理する個人スキル。
  「アルバイト管理」「アルバイト追加」「アルバイト探して」「医師バイト」「外勤」
  「非常勤」「シフト追加」「勤務先追加」「勤務完了」「先月のバイト集計」
  「バイトレポート」「baito」「locum」「moonlighting」「当直探して」
  「土日のバイト」のリクエストで必ず発動。
```

20 語以上のトリガー候補をカンマ区切りで並べています。これがいいか悪いかは設計判断ですが、私の運用では次の経験則を採用しています。

- **発動誤検知よりも、発動漏れの方が痛い**: 別表現でリクエストしたときに skill が起動しないと、毎回手動で呼び出すことになる
- **同一意味の語を、英日両方で並べる**: 「バイト」「locum」「moonlighting」など
- **専門用語と俗語の両方**: 「外勤」「土日のバイト」を両方入れる
- **発動を絞りたい場合は、否定的な条件をプロンプトに明記**: たとえば「臨床判断には使わない」「患者情報は扱わない」を本文側に書く

---

## どこまで skill 化すべきか

これは医師業務で「どこまでクリニカルパスを作るべきか」と同じ問いです。判断基準を提示します。

skill 化する:
- 月に 2 回以上繰り返している作業
- 手順が言葉で書ける(暗黙知が少ない)
- 失敗時のコストが小さい(リバーシブル)

skill 化しない:
- 一度きりの作業(別途 Claude に直接頼む)
- 文脈依存が極端に大きい(毎回判断が違う)
- 不可逆な操作(本番 DB 削除など)

迷ったら **「6 ヶ月後の自分が、同じ skill を読んで意味が分かるか」** で判断してください。SKILL.md は将来の自分への申し送り書です。

## scheduled-tasks の使い時

時間駆動が必要なのは次の条件のいずれかが立つときです。

- **見逃したくない情報を、定期的に拾う**(論文、ニュース、案件メール)
- **存在を忘れていても勝手に走ってほしい**(月次のメール整理、週次のレポート)
- **本人が起きていない時間に進めたい**(夜間のバッチ処理、朝の準備)

私の scheduled-tasks 登録例(抜粋):

- `ophthalmology-weekly-review`: 毎週月曜 07:00 JST(眼科主要ジャーナルの注目論文要約)
- `content-publishing-collect`: 毎朝 06:00 JST(Claude 関連ネタの収集)
- `content-publishing-weekly-summary`: 毎週日曜 20:00 JST(α 発信の週次レビュー)
- `trend-research`: 毎月 1 日 07:00 JST(ファッション・旅行・グルメのトレンドリサーチ)

cron 式は LOCAL タイムゾーンで評価されます。UTC ではないので、`0 7 * * *` と書けば日本ローカルの 7 時に動きます。

## ハイブリッド設計 — skill + scheduled-tasks

最も強力なパターンは、skill を scheduled-tasks に乗せることです。skill 単体ではキーワードで起動する人間が必要ですが、scheduled-tasks に登録すれば、本人不在で skill が走り続けます。

具体例として、`ophthalmology-weekly-review` skill は次の二重起動を持っています:

1. ユーザーが「眼科週次レビュー」と言ったらキーワード発動
2. scheduled-tasks `ophthalmology-weekly-review` で毎週月曜 07:00 JST に自動発動

両方とも同じ内部スクリプト(PubMed 巡回 + Notion 投入)を呼びます。設計判断としては:

- 平常時は (2) の自動発動が回る、本人は通知だけ受け取る
- 旅行や当直で skill 結果を見落とした週は、戻ってから (1) で手動再実行できる

このパターンは「定期回診と頓服指示」の構造に近いです。基本は予定通り回す、必要なときは追加投与する。

## 章末演習

自分が日常で繰り返している作業を 10 個書き出し、次の 3 つに分類してください。

- **キーワード起動向き**(skill 単体): 不定期、頼みたい瞬間が決まっていない
- **時間起動向き**(scheduled-tasks のみ): 定期、忘れても自動で走るべき
- **ハイブリッド向き**(skill + scheduled-tasks): 定期 + 任意の追加実行

分類できたら、各カテゴリの上位 1 つずつを実装してください。3 つの skill が稼働するだけで、月の余白時間が体感で増えます。
