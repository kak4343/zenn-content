## なぜこの 4 つだけ理解すれば十分か

Claude Code には機能が多くあります。が、個人インフラを動かすために最初に押さえるべきは、たった 4 つです。

- **skill** — 再利用可能な作業単位
- **scheduled-tasks** — 時間起動の自動化
- **hooks** — ツール実行前後の介入
- **MCP** — 外部サービス連携の標準プロトコル

この 4 つで「何を / いつ / 何の許可で / どこから取って」が全部記述できます。本章ではこれを、医師の業務に置き換えて説明します。

---

## 4 要素を医療業務に類比する

頭の整理として、それぞれを病院業務の何に対応させると分かりやすいかを書きます。あくまで類比で、厳密な対応ではありません。

- **skill = 標準診療プロトコル(クリニカルパス)**
  - 「この症状の主訴に対して、この順序で検査・治療を進める」というガイドライン化された手順
  - 一度作れば誰が回しても同じ結果に近づく
  - skill も同じです。`SKILL.md` に「いつ発動するか」「何をするか」を書いておけば、6 ヶ月後に自分で読み返しても、再現性のある作業として動く
- **scheduled-tasks = 定期回診 / 朝のラウンド**
  - 「毎週月曜 7 時に病棟を回って、注目患者の進捗を確認する」
  - 時間起動で、見落としを減らすための仕組み
  - scheduled-tasks も同じです。cron で `毎週月曜 07:00` と書けば、Claude が代わりに「論文ジャーナルを回って注目論文をピックする」「副業のメールから案件を抽出する」を勝手に走らせる
- **hooks = 院内ルール / 確認サイン**
  - 「このパスの処方は薬剤師のダブルチェックを通さないと出ない」「この検査オーダーは指導医のサインがないと走らない」
  - 行為の前に介入して、危険な操作を未然に止める
  - Claude Code の hooks は、Edit / Write / Bash などのツール呼び出しの前後で任意のスクリプトを走らせられる。患者情報を含むパスへの Edit を deny する設計が典型例(第 3 章 / 第 7 章で詳述)
- **MCP = 院外連携(他院紹介状 / 検査外注)**
  - 自院でやらない検査は外部に投げる。返ってきた結果をカルテに統合する
  - MCP も同じで、Notion / Gmail / Slack / カレンダー 等の外部サービスを、Claude から共通の作法で叩ける標準プロトコル

この 4 つを意識しておけば、新しい使い方を覚えるときに「これは skill にすべきか、scheduled-tasks にのせるか、それとも hooks で済むか、MCP の出番か」と頭の中で分類できるようになります。

## どこから入るのが最短か

「全部理解してから動かす」のは最悪のパターンです。USMLE 学習でも、全範囲を網羅してから過去問に入った人は、ほぼ全員落ちました。Claude Code も同じです。次の順で 1 つずつ触ってください。

1. **hooks から入る(初日)**
   - 患者情報や機密ファイルを deny する 6 行の設定だけを `settings.json` に追加する
   - これだけで「Claude が暴走しても、自分の臨床ディレクトリには手が出ない」状態になる
   - 後続の skill / scheduled-tasks を安心して触れる土台ができる
2. **skill を 1 つだけ作る(1 週目)**
   - 自分の業務外で繰り返しているタスクを 1 つ選び、`SKILL.md` だけ書く(スクリプトはあとで)
   - 例: 朝のメール一覧を要約する、論文を 200 字でまとめる
   - frontmatter の `description` が発動条件になることを体感する
3. **scheduled-tasks にのせる(2 週目)**
   - 上で作った skill を「毎朝 6 時に走らせる」など、時間起動にする
   - 朝起きると結果が Notion に届いている、という体験を 1 つ作る
4. **MCP は最後でいい(必要になってから)**
   - Notion API を叩きたい、Google Calendar から予定を取りたい、と思った時点で初めて MCP を調べる
   - 何もない段階で MCP から入ると、抽象度が高すぎて手が動かない

この順序は医師の臨床トレーニングの「シミュレーション → 観察 → 介助 → 執刀」とほぼ同じです。一番安全な層から段階的に責任を増やす。

## CLAUDE.md を整理するチェックリスト

`~/.claude/CLAUDE.md` は、Claude Code が起動時に必ず読む「横断ルール」です。私のものは現時点で約 100 行、次のような項目に絞っています。

- 言語と出力(日本語が既定、略語は初出でフル名併記)
- データ保護の不変条件(クラウドに送らないデータの定義)
- ファイル変更の作法(編集前に Read、永久削除しない)
- 並列タスクの上限(8GB Intel Mac なので 2 並列まで)
- iPhone 主体ワークフローの優先
- スキル横断ルール(SKILL.md の場所、Notion API トークンの保存先)
- 進め方の原則(Plan → 自己レビュー → 実行)

章末の演習として、自分の `CLAUDE.md` を 50 行以内に絞ってみてください。チェックリストは次の通り:

- 不変条件(絶対に守るルール)だけ書く。好みや細かい手順は別ファイルへ
- スキル個別の手順は `~/.claude/skills/<name>/SKILL.md` に分離
- データ保護関連のルールは最上段に置く(目に入りやすい場所)
- 並列タスクやマシン制約など、物理的な上限を明記する
- 1 行で 1 ルール、長文の説明は別 Markdown へリンクで逃がす

50 行以内に収まらなければ、ほぼ確実に「ルール」ではなく「手順」を書いてしまっています。手順は skill 側に移してください。
