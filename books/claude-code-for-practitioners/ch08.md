title: "医学論文の週次レビューを 7 分で終わらせる"
---

## この章の意図

ここまでの 7 章は安全層と原則の話が中心でした。本章からは「その安全層の上で、何が安全に走るのか」を具体例で示します。

題材として選んだのは、私が毎週月曜の朝に走らせている `ophthalmology-weekly-review` という skill です。眼科は私の専門領域ですが、本章で扱う設計パターンは、PubMed に索引されているあらゆる医学領域に移植できます。

---

## 何をしている skill か

毎週月曜 07:00 JST に scheduled-tasks から自動起動し、次の手順を踏みます。

1. 主要 8 ジャーナル(Ophthalmology / JAMA Ophthalmology / IOVS / AJO / BJO / Retina / Cornea / Journal of Glaucoma)と PubMed の最新 1 週間分を巡回
2. **predatory journals と medRxiv プレプリントを除外**
3. 注目論文 5〜7 本を選定
4. 各論文を日本語で要約(タイトル和訳・研究デザイン・主要結果・臨床的意義)
5. Notion Knowledge DB に投入(タグ: 論文 / Status: 未読 / Specialty: 専門領域 / Source URL 必須)
6. iPhone へ完了 push 通知

実行時間: 約 5-7 分。API コスト: 約 0.08 ドル/週。

私の朝のラウンドが終わるまでに、その週の「読むべき論文」が Notion に並んでいる、という状態を作っています。

---

## 設計判断 1: ジャーナル制限を最初から入れる

PubMed は便利ですが、検索結果には predatory journal も混ざります。skill 設計の最初の段階で、許可リストを置きました。

```python
ALLOWED_JOURNALS = [
    "Ophthalmology",
    "JAMA Ophthalmology",
    "IOVS",
    "American Journal of Ophthalmology",
    "British Journal of Ophthalmology",
    "Retina",
    "Cornea",
    "Journal of Glaucoma",
]
```

許可リスト方式の利点:

- **新規 predatory journal が出ても自動で除外**: 知らないジャーナル名は最初から通らない
- **アップデート容易**: 信頼するジャーナルが増えたら 1 行追加
- **ノイズが激減**: 検索結果の絞り込みが事前に効く

欠点も書きます。許可リストにない優れた論文を見逃すリスクがあります。私の場合、月 1 回 PubMed 全体を別 skill で巡回してリスト見直しに当てています。

---

## 設計判断 2: プレプリントを除外

medRxiv や bioRxiv に上がっているプレプリントは、査読を経ていません。臨床判断材料として扱うには早すぎます。

```python
EXCLUDED_SOURCES = ["medrxiv", "biorxiv", "preprints.org"]
```

除外理由を「skill の不変条件」として SKILL.md にも明記しています。

> 出典 URL 必須、predatory journal と medRxiv プレプリントは除外。

これは将来の自分への申し送り書です。半年後に「プレプリントも含めるか」と迷ったとき、なぜ除外したかを思い出せます。

---

## 設計判断 3: 要約フォーマットを 4 項目に固定

日本語要約は、毎回次の 4 項目構造で書きます。

1. **タイトル和訳**(原題と並列で表示)
2. **研究デザイン**(RCT / 観察研究 / メタアナリシス / 症例集積、サンプルサイズ)
3. **主要結果**(数値と統計的有意性)
4. **臨床的意義**(明日の外来で何が変わるか、変わらない場合はそう書く)

固定フォーマットの利点:

- **30 秒で読める**: スキャン読みで「自分の臨床に効くか」が判断できる
- **比較しやすい**: 複数論文を横並びで見ても、必要な情報の位置が同じ
- **AI が外しにくい**: 構造化されたプロンプトはハルシネーションが減る

特に「臨床的意義」の項目は、AI が美辞麗句で埋めがちな部分です。私のプロンプトには「変わらない場合はそう書け」と明示しています。

---

## 設計判断 4: 患者情報は絶対に扱わない

これは skill レベルの不変条件です。`ophthalmology-weekly-review` は次のものを **絶対に** 入力として受け取りません。

- 自分の症例
- 自分のカルテのコピー
- 自分が見た患者の検査画像
- 自分の所属病院のデータ

入力は **公開済みの査読論文のみ**。それ以外を入れる経路を最初から塞いでいます。第 3 章 / 第 7 章の deny-hook で、`patient*` / `clinical*` パスからの Read を全部止めているのも、この skill の不変条件と整合します。

---

## scheduled-tasks への登録

scheduled-tasks の MCP 経由で次のように登録しています。

```
taskId: ophthalmology-weekly-review
cronExpression: 0 7 * * 1   (毎週月曜 07:00、LOCAL タイムゾーン)
prompt: |
  以下のコマンドを実行してください:
  python3 ~/.claude/skills/ophthalmology-weekly-review/scripts/fetch_papers.py
  失敗時は Push 通知。
```

prompt 内で「失敗時の通知方針」を明記しているのは、scheduled-tasks のセッションが本人不在で動くからです。エラーで止まったまま気づかないのが最悪のパターンなので、失敗時の挙動を prompt 側で固定しています。

## カスタマイズ:他科への移植

本 skill のスケルトンは、ほぼそのまま他科に流用できます。書き換えが必要なのは次の 3 箇所だけ:

1. `ALLOWED_JOURNALS` の中身を自分の専門領域に
2. 要約プロンプトの「臨床的意義」を、自分の専門領域の判断軸に合わせる
3. Notion DB の Specialty multi_select の選択肢を増やす

たとえば「循環器週次レビュー」を作るなら、`ALLOWED_JOURNALS` を `Circulation` / `JACC` / `European Heart Journal` 等に差し替えるだけです。

## 章末演習

自分の専門領域で、本 skill のスケルトンを動かしてみてください。次の 3 ステップ:

**ステップ 1**: 自分が信頼するジャーナルを 5-8 本書き出す。許可リスト方式の母集団になる。

**ステップ 2**: 自分の専門領域用の「臨床的意義」プロンプトを 1 文書く。たとえば循環器なら「明日の外来でガイドラインを変えるべき内容か、変えないなら何か新しい知見か」のような問いに翻訳する。

**ステップ 3**: Notion Knowledge DB を作り、本 skill のスクリプト雛形を 1 つコピーして、ジャーナル名と「臨床的意義」プロンプトだけ書き換える。これで自分の専門領域の週次レビュー skill が動き始めます。
