## なぜ「階層化」なのか

医療安全の鉄則は、単一の防御線に頼らないことです。患者誤認を防ぐリストバンドが破れても、薬剤師のダブルチェックがある。それも抜けても、看護師の最終確認がある。一つの仕組みが落ちても、別の層が拾う。これを Swiss cheese model(スイスチーズモデル)と呼びます。

Claude Code の安全設計も同じで、私は次の 3 層で守っています。

- **Keychain(クレデンシャル層)** — そもそも API トークンを平文で持たない
- **hooks deny(行為層)** — 危険なツール呼び出しを実行前に止める
- **sandbox(実行層)** — Claude が動く範囲そのものを物理的に制限する

どれか一つで足りる、と思った瞬間に事故が起きます。3 層を全部かけてください。

---

## 第 1 層: Keychain

API トークンを `.env` ファイルや `~/.bashrc` に書いている人は、それを今すぐやめてください。`.env` は git に commit しやすく、`~/.bashrc` は `cat` で覗かれます。

macOS なら macOS Keychain、Linux なら `secret-tool`、Windows なら Credential Manager。いずれもユーザーログインに紐付いた暗号化ストアです。

macOS のコマンド例:

```bash
# 保存
security add-generic-password -s notion-api -a personal -w <token>

# 取り出し
security find-generic-password -s notion-api -a personal -w
```

私の運用では、Notion API トークン / Anthropic API キー / GitHub PAT などをすべて Keychain に置いて、Python スクリプトから `subprocess.run(["security", "find-generic-password", ...])` で取り出すようにしています。これで:

- ファイルシステム上にトークンが残らない
- スクリーンシェアでも漏れない
- バックアップに乗らない(`~/Documents` を Time Machine に取っても、Keychain は別領域)

トークンは漏れた瞬間に書き換えるしかない高価な資産です。だから「最初から触られない場所」に置く、というのが第 1 層です。

## 第 2 層: hooks deny

`~/.claude/settings.json` の `permissions.deny` 配列で、特定のツール呼び出しを未然に拒否できます。最小構成は次の通り:

```json
{
  "permissions": {
    "deny": [
      "Edit(**/*clinical*/**)",
      "Edit(**/*clinical*)",
      "Edit(**/*patient*/**)",
      "Edit(**/*patient*)",
      "Edit(**/*患者*/**)",
      "Edit(**/*患者*)",
      "Edit(**/*症例*/**)",
      "Edit(**/*症例*)",
      "Edit(**/*.dcm)",
      "Write(**/*clinical*/**)",
      "Write(**/*clinical*)",
      "Write(**/*patient*/**)",
      "Write(**/*patient*)",
      "Write(**/*患者*/**)",
      "Write(**/*患者*)",
      "Write(**/*症例*/**)",
      "Write(**/*症例*)"
    ]
  }
}
```

これは「ファイルパスにこれらの文字列が含まれる場合は、Edit / Write を拒否する」ルールです。Claude が「これは大事そうだから編集しません」と判断するのではなく、ツールレイヤーで物理的に止めます。Claude の善意に頼らないのが要点です。

パターン記法には注意が必要です。`permissions` のパスは gitignore 形式で評価され、`*` は同一階層内(`/` をまたがない)、`**` は再帰的にマッチします。そのため `clinical` を「パスのどこに現れても止めたい」なら、ディレクトリ用(`**/*clinical*/**`)とファイル名用(`**/*clinical*`)を 2 本セットで書きます。`Edit(*clinical*)` のように `**` を付けないと、カレントディレクトリ直下のファイル名にしかマッチせず、`~/clinical/notes.md` のようなパスはすり抜けます。特定ディレクトリだけを守るなら `Edit(~/clinical/**)` のようにホーム相対(`~/`)や絶対パス(`//`)で明示的にアンカーする方が確実です。

Bash 経由の `cp` / `mv` / `rm` まで止めるなら、PreToolUse hook + 自前スクリプトに切り替えます。スクリプトの全文は本書の `references/hook_recipes.md` 相当の章(後章)で提示しますが、要点は次の通り:

- `Edit | Write | MultiEdit | Bash` を matcher にする
- 標準入力で来る tool_input JSON の中の `file_path` / `command` を全パターン照合
- 1 件でもヒットしたら `hookSpecificOutput.permissionDecision` を `deny` にした JSON を返して終了(詳細と全文は第 7 章)

deny ルールは「一度書けば永続的に効く」のが価値です。1 ヶ月後に Claude のバージョンが上がっても、新しい skill を追加しても、このルールは生き続けます。

## 第 3 層: sandbox

最後の層は、Claude が動く土俵そのものを制限することです。次の選択肢があります。

- **dangerously-skip-permissions を使わない**: 研究コードでも個人作業でも、`--dangerously-skip-permissions` を恒常的に使わない。最初のうちは権限プロンプトが煩わしく感じますが、これが最後の人間の関門です
- **作業ディレクトリを限定する**: Claude を起動するときは、必ず該当プロジェクトのトップディレクトリで起動する。ホームディレクトリで起動しない。`cd ~/Projects/<repo>` してから `claude`
- **ネットワーク制限を検討する**: 自宅 / 職場の Wi-Fi にファイアウォール層で「Anthropic API と Notion API 以外には繋がない」設定を入れる(上級者向け)
- **Docker 化を検討する**: 機密性の高いプロジェクトは、Docker コンテナの中で Claude Code を動かす。ホスト OS のホームディレクトリには直接触れない

3 層目は、1 / 2 層目を補強する保険です。「自分は気をつけているから大丈夫」と思った日に事故は起きるので、物理的に範囲を狭めておきます。

## 章末演習

紙か Markdown に、自分のホームディレクトリの中で「AI に絶対触らせない 5 パス」を書き出してください。例えば:

- `~/clinical/` — 業務メモ全般
- `~/Documents/finance/private/` — 投資の生データ
- `~/.ssh/` — SSH 鍵
- `~/.aws/` — AWS クレデンシャル
- `~/Library/Application Support/<業務アプリ>/` — 機密データ

書き出したら、それを `permissions.deny` ルールに翻訳して `~/.claude/settings.json` に追記してください。それが本章の成果物です。
