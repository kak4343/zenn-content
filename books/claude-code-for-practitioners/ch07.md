## この章で書くこと

本章は本書で最も重要な章のひとつです。医療従事者が Claude Code を使う上で、最初に組むべき安全装置を、私の実運用設定そのままで公開します。

具体的には次の 3 段構成です。

- **第 1 段: 静的な deny ルール**(`settings.json` の `permissions.deny`)
- **第 2 段: 動的な PreToolUse hook**(シェルスクリプトでパターン照合)
- **第 3 段: 物理的な範囲制限**(Claude Code を呼ぶディレクトリの限定)

3 つを重ねれば、Claude が暴走しても臨床ディレクトリには手が出ない状態を作れます。

---

## 第 1 段: 静的な deny ルール

`~/.claude/settings.json` の `permissions.deny` 配列に、次のパターンを追加します。

```json
{
  "permissions": {
    "deny": [
      "Edit(**/*clinical*/**)", "Edit(**/*clinical*)",
      "Edit(**/*patient*/**)", "Edit(**/*patient*)",
      "Edit(**/*患者*/**)", "Edit(**/*患者*)",
      "Edit(**/*症例*/**)", "Edit(**/*症例*)",
      "Edit(**/*case_report*/**)", "Edit(**/*case_report*)",
      "Edit(**/*.dcm)",
      "Edit(**/*EHR*/**)", "Edit(**/*EHR*)",
      "Edit(**/*カルテ*/**)", "Edit(**/*カルテ*)",
      "Write(**/*clinical*/**)", "Write(**/*clinical*)",
      "Write(**/*patient*/**)", "Write(**/*patient*)",
      "Write(**/*患者*/**)", "Write(**/*患者*)",
      "Write(**/*症例*/**)", "Write(**/*症例*)",
      "Write(**/*case_report*/**)", "Write(**/*case_report*)",
      "Write(**/*.dcm)",
      "Write(**/*EHR*/**)", "Write(**/*EHR*)",
      "Write(**/*カルテ*/**)", "Write(**/*カルテ*)",
      "Read(**/*clinical*/**)", "Read(**/*clinical*)",
      "Read(**/*patient*/**)", "Read(**/*patient*)",
      "Read(**/*患者*/**)", "Read(**/*患者*)"
    ]
  }
}
```

ポイント:

- **`**` を必ず付ける**: パスは gitignore 形式で、`*` は `/` をまたがない。`Edit(*clinical*)` ではカレント直下のファイル名しか止まらず、`~/clinical/notes.md` がすり抜ける。ディレクトリ用(`**/*clinical*/**`)とファイル名用(`**/*clinical*`)を 2 本セットで書く(第 3 章参照)
- **英日両方**: `patient` と `患者`、`clinical` と `カルテ` をペアで入れる。コード上のファイル名は英語が多いが、自分のローカルメモは日本語のことがある
- **拡張子と内容語の両方**: `.dcm`(DICOM 医療画像)、`EHR`(電子カルテシステム)など、拡張子・略語ともにカバー
- **Read もブロック**: 編集だけでなく読み取りもブロックすると、要約や転載のリスクが減る

これだけでも 80% の事故は防げます。残り 20% を埋めるのが第 2 段です。

---

## 第 2 段: 動的な PreToolUse hook

静的な deny ルールでカバーできない 2 つの抜け穴があります。

- Bash 経由の `cp` / `mv` / `rm` でファイルを動かす操作
- `MultiEdit` で同時に複数ファイルを操作する場合の照合漏れ

これらを止めるには PreToolUse hook + 自前スクリプトに切り替えます。

`~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit|Bash|Read",
        "hooks": [
          {
            "type": "command",
            "command": "/Users/<USERNAME>/.claude/hooks/deny-clinical.sh"
          }
        ]
      }
    ]
  }
}
```

`~/.claude/hooks/deny-clinical.sh`:

```bash
#!/bin/bash
# Block any tool call touching clinical/patient paths.
set -e
INPUT=$(cat)

HIT=$(echo "$INPUT" | /usr/bin/python3 -c '
import json, sys, re
data = json.load(sys.stdin)
ti = json.dumps(data.get("tool_input", {}), ensure_ascii=False)
patterns = [
    r"clinical",
    r"patient",
    r"患者",
    r"症例",
    r"case_report",
    r"\.dcm",
    r"EHR",
    r"カルテ",
]
for p in patterns:
    if re.search(p, ti, re.IGNORECASE):
        print(p)
        sys.exit(0)
')

if [ -n "$HIT" ]; then
  cat <<JSON
{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"clinical pattern hit: $HIT"}}
JSON
  exit 0
fi
echo '{}'
```

`chmod +x ~/.claude/hooks/deny-clinical.sh` を忘れずに。

**重要(現行スキーマ)**: PreToolUse hook でツール呼び出しを止めるには、`hookSpecificOutput` オブジェクトに `hookEventName: "PreToolUse"` と `permissionDecision: "deny"` を入れて返します。`permissionDecision` の有効値は `allow` / `deny` / `ask`。旧来の `{"decision":"deny"}` という形は **無視され、ツールがそのまま実行されてしまう** ので使わないこと(`decision` を使う場合の停止値も "deny" ではなく "block")。JSON を組み立てずに確実に止めたいだけなら、スクリプトを `exit 2` で終了させる方法もあります。

このスクリプトの要点:

- **tool_input の全フィールドを JSON 文字列化してから照合**: `file_path` だけでなく `command` 内のパスや、`MultiEdit` の `edits` 配列内のパスも全部チェック
- **大文字小文字無視**: `Patient`, `PATIENT`, `patient` すべてヒット
- **deny 判定したら即終了**: 後続の判定はしない、確実に止める

## 第 3 段: 物理的な範囲制限

最後の層は、そもそも Claude を「臨床ディレクトリのあるホームディレクトリの上位」では起動しないことです。

私の運用ルール:

- `~` 直下で Claude Code を起動しない(`cd ~` の状態で `claude` と打たない)
- 必ずプロジェクトディレクトリに `cd` してから起動: `cd ~/Projects/<repo>` → `claude`
- 臨床メモは `~/clinical/` 配下に集約、それ以外には置かない
- 機密性の高い作業は Docker コンテナ内で行う(コンテナはホスト OS の `~/clinical/` をマウントしない)

これは Claude のせいではなく、自分のディシプリンの問題です。3 層目をサボると、1 / 2 層目が破られたときに防げません。

---

## 運用上の注意

### 注意 1: hook が遅延を生む

PreToolUse hook はツール呼び出しのたびに動くため、わずかに遅延します。私の環境では 1 ツール呼び出しあたり 100-200ms 程度。実用上は無視できますが、巨大な MultiEdit を多用する作業では体感できる場合があります。

### 注意 2: deny の理由はログに残る

deny したときに `permissionDecisionReason` フィールドに書いた内容は、Claude のセッションログに残ります。患者情報そのものを理由に書かないこと。「clinical pattern hit: patient」のような汎用的な文字列に留める。

### 注意 3: チームで共有するときの注意

GitHub に `settings.json` を上げるときは、`<USERNAME>` を `${HOME}` や相対パスに書き換える。実パスをコミットしないこと。

## 章末演習

次の 3 ステップで自分の運用に落とし込んでください。

**ステップ 1**: 第 1 段の `permissions.deny` を `~/.claude/settings.json` に追加し、1 週間運用する。誤検知(deny されすぎ)があれば、ログを見て調整。

**ステップ 2**: 1 週間後、第 2 段の `deny-clinical.sh` を追加。`patient` の照合パターンを自分の業務環境に合わせて拡張(専門分野の略語、自院のシステム名など)。

**ステップ 3**: 2 週間後、第 3 段の物理的範囲制限を運用ルール化。`~/.zshrc` の `prompt` に「現在のディレクトリが clinical 配下かどうか」を表示する 1 行を足すと、視覚的に気づきやすくなる。
