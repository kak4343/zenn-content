## はじめに

私は USMLE Step 1 と Step 2 CK を独学で取得した日本人医師です。Step 2 CK のスコアは 231、Pass ライン(Borderline)から逆転で取りました。渡米予定はなく、その代わりに、Claude Code で自分の業務外の生活インフラ(論文管理、USMLE 学習、副業の運用、投資の意思決定支援)をほぼ全部自動化する方向に舵を切りました。

本書はそのプロセスで作った skill と、設計判断の裏側を、章単位で公開していくものです。本書の対象読者は「Claude Code を個人インフラ全体に効かせたい」エンジニア・医師・研究者です。

第 1 章では、なぜ医師という職業の頭の使い方が Claude Code と相性が良いのか、その構造的な理由を書きます。結論を先に書きます。**医師の訓練の中核 ─ 不確実性下での意思決定、再現性へのこだわり、データ保護への執着 ─ は、そのまま Claude Code の運用設計に転用できます。** むしろ転用しないのは損です。

---

## 1. 臨床判断と AI への指示は同じ構造をしている

外来診療の意思決定は、おおまかに次の手順で進みます。

- 主訴を限定する(「胸が痛い」を、急性 / 慢性、左 / 右、放散の有無で絞る)
- 鑑別を上げる(限定された主訴に対し、ありうる疾患を列挙する)
- 優先順位をつける(緊急度・頻度・治療可能性で並べる)
- 検査・問診で絞る
- 治療方針を決め、結果を評価し、必要なら次の手を打つ

Claude Code に仕事を任せるときの思考プロセスは、これとほぼ同じです。

- タスクを限定する(「論文を要約して」ではなく「過去 1 週間に出た網膜剥離 RCT を 200 字で」)
- 解法の候補を上げる(skill を作る / scheduled-tasks に乗せる / 一回の対話で済ます)
- 優先順位をつける(再利用性・コスト・確実性)
- 実装して結果を評価する
- 次のタスクへ転用するか、切り捨てるかを決める

違いは 1 点だけです。臨床は生身の患者を扱うため、間違いの代償が大きい。一方 Claude Code は再現可能な実行環境で動くため、間違ってもロールバックできます。この「コストが圧倒的に低い実験場」の中で、自分の判断プロセスを試行錯誤できるのが、Claude Code を使うことの最大の利点です。

医師にとってこれは、新しいスキルセットを獲得する話ではなく、既に持っている思考様式を、別のドメインで再使用するだけです。

---

## 2. 再現性こそが両者を結ぶ

医学が依拠する EBM(evidence-based medicine、根拠に基づく医療)は、要するに「同じ条件なら同じ結論に至るべきだ」という再現性への信念です。ガイドラインも同じ哲学に立っています。USMLE は「同じ問題に対して、同じ最良の解を選べるか」を執拗に試す試験です。私が 900 時間で Borderline から Pass に逆転できたのも、再現性のある勉強法に絞り、当たり外れの大きい教材を外したからでした。

Claude Code の skill 設計は、これと同じ哲学で動いています。skill とは「同じ入力なら同じ出力を返す、最小単位の作業」です。skill を JSON や Markdown で記述しておけば、6 ヶ月後に自分で読み返しても、他人が読んでも、挙動が再現する。再現しないなら、それは skill ではなく単発の対話作業です。

「これは skill 化すべきか、その場で頼むべきか」の判断は、医師の言葉に翻訳すれば「これはガイドラインに載せるべきか、症例ごとの判断に委ねるべきか」と同じ問いです。私は前者の問いに毎日答える訓練を受けてきたので、後者の判断にも自然に応用できます。逆に言えば、エンジニアの中でも「再利用しすぎて DRY 違反になる」とか「単発でいいのに過剰に skill 化する」とかの過剰反応をしないで済みます。

## 3. データ保護への執着が良い実装を生む

医師の最も厳しい職業習慣は「患者情報を漏らさない」です。電子カルテのスクリーンショットを SNS にあげる医師はいません。臨床メモを自宅 PC に持ち帰る医師もいません(少なくとも、自分が訓練された範囲では)。

この習慣は、そのまま Claude Code の hooks による deny ルール設計に転用できます。

- `~/clinical/` 以下のファイルは AI に Edit させない
- `*患者*` `*症例*` `*.dcm` で始まる / 終わるパスは Read もさせない
- Bash 経由で `cp` `mv` するときも、同じパターンを deny する

この設計は、第 3 章(安全設計の 3 階層)と第 7 章(患者情報を絶対外に出さない hooks レシピ)で詳述します。要点は、deny ルールは一度設定すれば永続的に動く safeguard だということ、そして「漏らさないために、最初から見せない」という発想が、医師にとって自然だということです。

データ保護への執着は、結果として実装品質を底上げします。「このパスにだけは触らせない」と決めれば、それを破ろうとする実装は最初から書きません。実装が消極的に保守的になるのではなく、設計の初期段階で安全な範囲を確定させるからです。

## 4. 最初に Claude Code を使うべき 3 つの場面

医師が初めて Claude Code を触るなら、私は次の 3 つの場面を勧めます。いずれも患者情報をまったく扱わない領域です。

(a) **USMLE / 専門医試験の学習支援**:過去問の弱点分析、Anki カードの自動生成、解説テキストの構造化。私自身、Step 2 CK で Borderline からの逆転に必要だった「弱点領域の特定 → 該当領域の高頻度問題への集中投下」の半分は、自前の skill で機械化しました。

(b) **文献レビューの定期化**:私は毎週月曜 7:00 JST に、眼科主要ジャーナルと PubMed から注目論文 5〜7 本を要約して Notion に投入する skill を回しています。所要時間は 7 分、API コストは週 0.08 ドル。第 8 章で全実装を公開します。

(c) **自分のワークフロー自動化**:カレンダー整理、メール仕分け、ToDo 管理、Notion DB への構造化投入。臨床業務に直接触らないが、業務外時間を圧縮する用途です。

この 3 つから入れば、患者情報に触れずに Claude Code の安全な使い方を体得でき、その後で第 7 章の hooks 設計に進めば、より踏み込んだ用途にも自然に拡張できます。

## 章末演習

ここまで読んで、自分にも Claude Code が回せそうだと感じたら、次の演習に取り組んでみてください。

1. 自分の業務外(家事・学習・副業)の 1 日分のタスクを 30 分単位で書き出す
2. 各タスクについて、「Claude Code に任せられるか」を Yes / No / 一部 で判定する
3. Yes と「一部」のリストから、最初に着手する 3 つを選ぶ
4. 選んだ 3 つに対し、第 2 章(skill / scheduled-tasks / hooks / MCP)を読みながら、どの構成要素を使うかを決める

この演習の答えに「これは患者情報に触れる」が混ざっていたら、その項目はリストから外してください。理由は第 3 章と第 7 章で詳述します。
